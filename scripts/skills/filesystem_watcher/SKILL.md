## What It Does

The filesystem watcher automates the **Capture → Triage** step of the vault workflow. It continuously monitors `/Inbox` and processes any files that appear there.

When a new file is detected:

1. The file is **moved** from `/Inbox` to `/Needs_Action`.
2. A **task note** is created in `/Needs_Action` using the convention `YYYY-MM-DD Task - <filename>.md`.
3. The task note is pre-populated with: date received, original location, destination path, owner (unassigned), status (pending), and a standard review checklist.

**Behaviour details:**
- Polls `/Inbox` every 10 seconds.
- Hidden files (names starting with `.`) are skipped.
- If a file with the same name already exists in `/Needs_Action`, a timestamp suffix is appended to avoid overwriting.
- Does not read or interpret file contents — moves and logs only.

## How to Run It

**Prerequisite:** Python 3 installed.

**WSL / Linux:**
```bash
python /mnt/d/AI-EMPLOYEE-VAULT/scripts/filesystem_watcher.py
```

**Windows (CMD or PowerShell):**
```powershell
python D:\AI-EMPLOYEE-VAULT\scripts\filesystem_watcher.py
```

Press `Ctrl+C` to stop cleanly.

**Sample output:**
```

Watching: /mnt/d/AI-EMPLOYEE-VAULT/Inbox
Polling every 10 seconds. Press Ctrl+C to stop.

[09:15:32] Detected: report.pdf
  [move]  Inbox -> Needs_Action/report.pdf
  [note]  Created task note: 2026-03-24 Task - report.pdf.md
```

## When to Use It

| Situation | Use Watcher? |
|-----------|-------------|
| Files arrive in `/Inbox` frequently (attachments, exports, uploads) | Yes |
| You want triage to happen automatically without manual intervention | Yes |
| Running as a background/CI process while the vault is active | Yes |
| One-off manual triage of a small number of items | No — triage manually per [[Company_Handbook]] |
| Vault is idle with no expected inputs | No — stop the watcher |

---

## Limitations

- Task notes generated by the watcher are stubs — **Owner** and **Desired Outcome** must be filled in before the item is actionable.
- The watcher only processes `/Inbox`; it does not scan `/Needs_Action` or any other folder.
- No retry logic if a file move fails (e.g. due to a permissions error).
